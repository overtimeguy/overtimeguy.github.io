## 인벤토리 시스템 마스터

인벤토리 시스템의 1차 구현이 일단락됐다.

처음에는 장비 아이콘을 목록에 보여주고, 누르면 상세 정보가 뜨는 정도를 생각했다. 그런데 장착, 프리셋, 캐릭터, 강화, 합성, 제련, 룬, 장비 초기화까지 하나씩 연결하고 나니 사실상 작은 게임 하나가 되었다. 화면 하나 안에서 **서버 데이터, 성장 규칙, 재화, 선택 상태, 팝업, 로컬 저장, 로컬라이제이션**이 전부 만난다.

기능이 많아진 만큼 미래의 내가 다시 봤을 때도 구조를 이해할 수 있도록, 이번 글은 평소 개발일지보다 훨씬 자세하게 남기기로 했다. 회고이면서 현재 구현의 명세서이고, 문제가 생겼을 때 찾아보는 위키이기도 하다.

> 이 문서는 2026년 7월 10일의 로컬 프로젝트 코드를 기준으로 작성했다. 기억이나 계획이 아니라, 실제 클라이언트 코드와 PlayFab CloudScript, Title Data JSON을 다시 대조한 결과다.

---

## 한눈에 보는 완성 범위

| 영역 | 구현된 기능 |
|---|---|
| 장비 인벤토리 | 서버 장비 표시, 부위 필터, 3종 정렬, 장착 표시, 상세 팝업 |
| 캐릭터 인벤토리 | 보유·미보유 캐릭터 통합 표시, 잠금, 장착, 강화, 고유능력 단계 |
| 장비 상세 | 이름, 4종 능력치, 장착품 대비 증감, 등급 스킬, 룬 3개, 강화 |
| 장착 | 무기·투구·방패·갑옷·장갑·신발 6부위, 유효성 자동 복구 |
| 프리셋 | 이름 변경, 현재 장착 저장, 프리셋 전환, 빈 슬롯 보존 |
| 합성 | 메인·재료 선택, 자동 선택, 잠금 표시, 성장 수치 미리보기, 서버 합성 |
| 제련 | 장비·룬 선택, 룬 잠금, 비용 계산, 최초 감정, 등급 상승 추첨 |
| 장비 초기화 | 환급 미리보기, 강화 재화 환급, 임시장비 지급, 전체 새로고침 |
| 공통 안정성 | 전역 입력 차단, 중복 요청 방지, 로컬 캐시 패치, 성공 응답의 캐시 패치 실패 시 전체 재로드 |
| 표현 계층 | 등급 이미지, 장비·룬·캐릭터 데이터, k/m 숫자 표기, 다국어 토스트 |

이 시스템을 관통하는 원칙은 하나다.

> **서버가 소유권과 성장 결과를 결정하고, 클라이언트는 서버에서 받은 데이터를 빠르게 캐시하여 모든 화면이 같은 상태를 바라보게 한다.**

## 전체 구조

현재 데이터의 큰 흐름은 다음과 같다.

```text
PlayFab
├─ Catalog       : 장비·캐릭터의 고정 정의
├─ Title Data    : 강화·합성·제련·캐릭터 성장 밸런스
├─ UserInventory : 실제 소유 아이템, 인스턴스 상태, 가상 재화
└─ CloudScript   : 성장 검증과 서버 변경
        ↓
ServerManager
├─ ServerAuthModule
├─ ServerCatalogModule
├─ ServerTitleDataModule
├─ ServerCurrencyModule
└─ ServerInventoryModule
        ↓  Ready / OnServerDataLoaded / Refreshed
UI
├─ 기본 장비·캐릭터 인벤토리
├─ 장착 슬롯·프리셋
├─ 장비·캐릭터 상세 팝업
├─ 합성 인벤토리와 선택 컨트롤러
├─ 제련 인벤토리와 룬 컨트롤러
└─ 장비 초기화 팝업
```

`ServerManager.Ready`는 모듈 오브젝트가 준비되었다는 신호다. 실제 서버 데이터까지 준비된 시점은 Title Data, Catalog, Inventory가 모두 `Loaded`가 된 뒤 발생하는 `OnServerDataLoaded`다. 두 이벤트를 구분한 이유는 초기화 순서 때문에 생기는 간헐적인 null 문제를 막기 위해서다.

최초 화면 구성은 `InitialServerUiLoadController`가 한 번만 수행한다. 이후 서버 변경은 `Inventory.Refreshed`, 장착 변경은 `EquipmentLoadoutController.Refreshed`, 재화 변경은 `ServerCurrencyModule.Refreshed`를 통해 각 화면으로 전파된다.

## 장비 데이터 한 개가 가진 것

런타임에서 장비 하나는 `EquipmentData`로 표현된다.

| 값 | 의미 |
|---|---|
| `ItemId` | 카탈로그 장비 종류. 예: `eq_weapon_003` |
| `ItemInstanceId` | 플레이어가 실제로 가진 장비 한 개의 고유 ID |
| `EquipmentType` | Weapon, Helmet, Shield, Armor, Gloves, Boots |
| `CatalogIndex` | 같은 부위 안에서의 카탈로그 순번 |
| `IsTemporary` | 초기화 환급용 임시장비 여부 |
| `Grade` | 0~5 등급 |
| `Level` | 현재 강화 레벨. UI에는 `+1`과 같은 형태로 표시 |
| `Runes` | 3개 룬 슬롯의 `RuneId`, 등급, 슬롯 인덱스 |
| `SourceItemInstance` | PlayFab 원본 인스턴스 |

장비 인스턴스의 등급과 레벨은 PlayFab CustomData의 `state`에, 룬은 `r0`, `r1`, `r2`에 저장된다. 클라이언트는 이를 `EquipmentDataParser`로 읽고, 서버 작업 성공 후에는 같은 포맷으로 로컬 캐시도 다시 만든다.

고정 데이터와 표현 데이터는 분리했다.

- Catalog: 기본 능력치와 레벨 성장치
- Title Data: 등급별 최대 레벨, 비용, 배율
- GlobalGameData: 등급 배경, 배지, 장비·룬·캐릭터 이미지와 로컬라이제이션 텍스트
- EquipmentData: 특정 플레이어가 가진 한 장비의 현재 상태

덕분에 아이콘이나 텍스트를 바꿔도 서버 상태를 건드릴 필요가 없고, 기존 스키마와 고정 슬롯 수 범위 안의 수치 밸런스는 UI 프리팹을 수정하지 않고 바꿀 수 있다.

## 서버 데이터가 화면에 뜨기까지

로그인 뒤 Title Data, Catalog, Inventory 모듈은 서로의 완료를 기다리지 않고 독립적으로 로드를 시작한다. 따라서 아래 세 작업의 완료 순서는 보장되지 않는다.

1. `ServerTitleDataModule`이 강화·합성·제련·캐릭터 밸런스를 읽는다.
2. `ServerCatalogModule`이 장비와 캐릭터 정의를 읽는다.
3. `ServerInventoryModule`이 실제 인벤토리와 가상 재화를 읽고, 원본 아이템을 `Items`, 장비를 `EquipmentItems`, 캐릭터를 `CharacterItems`로 구성한다.
4. 세 모듈이 모두 `Loaded`가 되면 `OnServerDataLoaded`가 한 번 발생한다.
5. 초기 UI가 장착 슬롯, 장비 그리드, 캐릭터 그리드, 전투력 요약을 그린다.

인벤토리 로드는 일시적인 네트워크 오류에 대해 최대 3회 재시도한다. 캐릭터가 하나도 없다면 기본 캐릭터 지급 후 0.5초 간격으로 다시 확인하고, 디버그 모드에서 장비가 비어 있다면 테스트 장비도 별도 흐름으로 지급한다. 기본 지급과 디버그 시딩을 분리해 둔 덕분에 실제 유저 데이터와 개발용 데이터가 섞이지 않는다.

그리드는 `EnhancedGrid`를 사용한다. 필터나 정렬이 바뀌면 네트워크를 다시 호출하지 않고 이미 로드된 `EquipmentItems`만 재구성한다. 갱신 직전 스크롤 속도를 0으로 만들고, 마지막 행은 `null` 셀로 채운 뒤, 프레임 끝에 원래 MovementType을 복구한다. 빠르게 필터를 바꿀 때 스크롤이 튀거나 마지막 줄이 찌그러지는 문제를 이 과정으로 정리했다.

## 기본 장비 인벤토리

### 필터

필터는 다음 7개다.

- 전체
- 무기
- 투구
- 방패
- 갑옷
- 장갑
- 신발

필터 버튼은 자신이 어떤 `InventoryEquipmentFilter`를 담당하는지만 알고, 실제 목록 갱신은 `InventoryFilterGroup`과 `InventoryController`가 담당한다. 현재 선택된 필터는 항상 하나이며, 버튼의 선택 비주얼도 그룹이 한 번에 갱신한다.

### 정렬

정렬 버튼은 다음 순서로 순환한다.

```text
등급 순 → 종류 순 → 레벨 순 → 등급 순
```

1차 정렬 기준은 명확하다.

| 모드 | 1차 기준 |
|---|---|
| 등급 순 | 등급 내림차순 |
| 종류 순 | 무기 → 투구 → 방패 → 갑옷 → 장갑 → 신발 |
| 레벨 순 | 레벨 내림차순 |

동률이면 등급, 장비 종류, ItemId, 레벨, InstanceId를 차례로 사용하여 결과가 매번 흔들리지 않게 한다.

임시장비 후순위는 **전체 목록의 절대 마지막**이 아니다. 앞선 정렬 키가 같은 구간에서 ItemId를 비교할 때 일반 장비 뒤에 놓인다. 따라서 종류 순에서도 높은 등급 임시장비가 낮은 등급 장비 아래로 떨어지는 일은 없다.

```text
종류 → 등급 → 일반/임시 → ItemId → 레벨 → InstanceId
```

### 셀 표현

`InventoryView` 하나가 다음을 모두 표현한다.

- 등급별 배경
- 장비 아이콘
- `+Level`
- 장착 중인 장비의 `E` 표시
- 선택 체크 표시
- 사용할 수 없는 장비의 잠금 표시

같은 셀을 기본 인벤토리, 장착 슬롯, 합성, 제련, 초기화 미리보기에서 재사용한다. 데이터와 클릭 콜백만 다르게 주입하기 때문에 화면마다 장비 표시 규칙이 달라지는 문제를 줄였다.

일반 장비를 누르면 장비 상세 팝업이 열린다. 팝업이 열려 있는 동안 인벤토리 CanvasGroup의 입력을 막고, 팝업이 닫히면 입력과 현재 보이는 셀을 다시 갱신한다.

## 장착과 프리셋

장착은 6부위 각각 하나의 `ItemInstanceId`를 로컬에 저장한다.

```text
Weapon / Helmet / Shield / Armor / Gloves / Boots
```

장착 데이터는 PlayerPrefs 기반이지만, 매번 서버 인벤토리 캐시와 대조한다. 저장된 장비가 합성 재료로 소모되었거나 서버에서 사라졌다면 해당 슬롯만 비운다. 다른 정상 슬롯까지 초기화하지 않는다. 타입이 잘못되었거나 임시장비가 들어온 경우도 같은 방식으로 복구한다.

장착 슬롯 자체도 `InventoryView`이므로 누르면 동일한 장비 상세 팝업을 연다. 장비가 바뀌면 인벤토리의 `E` 표시, 장착 슬롯, 캐릭터 총 능력치, 전투력 요약이 함께 갱신된다.

프리셋은 이름과 실제 장착 데이터를 분리해 저장한다.

- 프리셋 이름·선택 인덱스: `PresetPrefs_v1`
- 6부위 장착 데이터: `EquipmentPresetPrefs_v1`

프리셋 전환 직전에는 현재 장착을 이전 프리셋에 저장하고, 새 프리셋을 검증한 뒤 적용한다. 장비를 개별 장착하거나 해제해도 현재 프리셋에 자동 반영된다.

여기서 중요한 규칙은 **빈 프리셋은 진짜 빈 프리셋**이라는 것이다. 비어 있는 슬롯을 임의 장비로 채우지 않는다. 저장된 장비 하나가 사라졌다면 그 슬롯만 비우고 나머지는 그대로 적용한다.

## 장비 상세 팝업

장비를 선택하면 다음 정보가 한 번에 갱신된다.

- 장비 이미지와 이름
- 공격력, HP, 방어력, 치명타 확률
- 같은 부위의 현재 장착 장비 대비 증가·감소량
- 장착·해제 버튼
- 단일 강화·일괄 강화 버튼과 비용
- 등급별 스킬 5개
- 룬 3개와 선택된 룬의 상세 정보
- 장비 초기화 팝업 진입 버튼

능력치 차이는 증가면 파란색 `▲`, 감소면 빨간색 `▼`로 표시한다. 이미 장착한 장비이거나 비교할 같은 부위 장비가 없다면 숨긴다. 표시상 0이 되는 차이도 보여주지 않는다.

장비 능력치는 현재 다음 식을 사용한다.

```text
기본 능력치 = Catalog 기본값 × 등급 배율²
성장 능력치 = 레벨당 성장값 × 현재 레벨 × 등급 배율
최종 능력치 = 기본 능력치 + 성장 능력치
```

등급 스킬은 1~5등급에서 차례로 열린다. `GlobalGameData`가 장비별 설명을 보유하고, 현재 등급과 요구 등급을 비교해 활성·비활성 이미지를 바꾼다.

### 장비 상세의 룬

장비의 룬은 슬롯 껍데기와 룬 데이터가 분리되어 있다. 팝업 진입 시 항상 0번 룬을 선택하고, 이후에는 한 번에 하나만 선택한다. 같은 룬을 다시 눌러도 선택 해제되지 않는다.

선택된 룬은 다음 정보를 `GlobalGameData`에서 가져온다.

- 등급 배지 이미지
- 일반·고급·레어·에픽·유니크·전설 등급명
- 룬 이름
- 현재 등급에 맞는 룬 설명

즉 룬의 `RuneId`와 등급만 서버 인스턴스에 저장하고, 이름과 설명과 이미지는 공용 데이터에서 해석한다.

## 캐릭터 인벤토리

캐릭터 인벤토리는 소유 목록만 보여주지 않는다. `GlobalGameData.CharacterVisuals`의 순서대로 모든 캐릭터의 껍데기를 먼저 만들고, 서버에서 소유한 캐릭터 데이터를 합친다. 덕분에 미보유 캐릭터도 잠긴 상태로 같은 자리에 유지된다.

캐릭터 표시 모델은 서버 데이터가 불완전할 때 아이콘 셸만 유지하고 레벨·조각 같은 불안정한 값을 숨길 수 있게 설계했다. 다만 현재 정상 자동 초기 로드 경로는 전체 서버 데이터가 준비된 뒤 `ApplyData`를 호출하므로, 로딩 도중 셸을 먼저 그리지는 않는다.

저장된 장착 캐릭터가 더 이상 유효하지 않으면 첫 번째 소유 캐릭터를 자동 장착한다. 캐릭터는 장비처럼 해제 상태를 만들지 않고 다른 캐릭터로 교체한다.

고유능력 패널은 다음 우선순위로 대상을 정한다.

```text
현재 장착 캐릭터 → 선택 캐릭터 → 첫 소유 캐릭터 → 첫 표시 캐릭터
```

여기서 선택 캐릭터 우선순위는 외부에서 `SelectCharacter`를 호출했을 때만 사용된다. 현재 그리드 클릭은 상세 팝업을 바로 열기 때문에 정상 배선에서는 사실상 장착 캐릭터 → 첫 소유 캐릭터 → 첫 표시 캐릭터 순서로 동작한다.

캐릭터 그리드에는 별도 필터나 정렬 버튼이 없다. `CharacterVisuals`에 등록된 순서를 따르고, 그 목록에 없는 소유 캐릭터만 뒤에 붙인다. 미보유 캐릭터도 상세 팝업은 열 수 있지만 장착과 강화 버튼은 숨긴다.

고유능력의 4개 단계는 캐릭터 레벨 1, 3, 6, 10에서 활성화된다.

캐릭터 상세에서는 캐릭터 자체 능력치에 현재 장착한 6부위 장비 능력치를 더해 총 전투력을 계산한다. 다른 캐릭터를 보고 있다면 현재 장착 캐릭터와의 능력치 차이도 표시한다.

현재 전투력 식은 다음과 같다.

```text
공격 기여 = 공격력 × 6 × (1 + 치명타 확률)
생존 기여 = HP × 8 + 방어력 × 10
전투력 = (공격 기여 + 생존 기여) × 1110.42
```

캐릭터 강화에는 캐릭터 조각을 사용한다. 최대 레벨은 10이며, 현재 레벨별 요구량은 다음과 같다.

| 현재 레벨 | 다음 레벨 요구 조각 |
|---:|---:|
| 1 | 20 |
| 2 | 50 |
| 3 | 100 |
| 4 | 200 |
| 5 | 300 |
| 6 | 450 |
| 7 | 600 |
| 8 | 750 |
| 9 | 950 |

서버는 실제 캐릭터 인스턴스의 조각 수를 검사하고 차감한 뒤 레벨을 올린다.

## 강화

장비 강화는 확률이 없다. 조건과 재화가 맞으면 항상 레벨이 오른다.

| 등급 | 이름 | 최대 레벨 | 1회 골드 | 1회 강화석 | 성장 배율 |
|---:|---|---:|---:|---:|---:|
| 0 | 일반 | 20 | 100 | 1 | 1.00 |
| 1 | 고급 | 40 | 160 | 2 | 1.15 |
| 2 | 레어 | 80 | 250 | 3 | 1.35 |
| 3 | 에픽 | 120 | 380 | 5 | 1.60 |
| 4 | 유니크 | 160 | 560 | 8 | 2.00 |
| 5 | 전설 | 200 | 800 | 13 | 2.50 |

단일 강화는 1레벨만 올린다. 일괄 강화는 다음 네 값 중 가장 작은 횟수만큼 진행한다.

- 현재 등급에서 남은 레벨 수
- 요청에서 지정한 최대 횟수
- 보유 골드로 가능한 횟수
- 보유 강화석으로 가능한 횟수

현재 UI의 일괄 강화는 별도의 횟수 상한을 보내지 않으므로, 재화가 허용하는 범위에서 최대 레벨까지 올린다. 한 번도 강화할 수 없다면 `CURRENCY_NOT_ENOUGH`로 실패한다.

임시장비와 최대 레벨 장비는 강화할 수 없다.

## 합성

합성은 장비의 등급을 한 단계 올리는 기능이다. 성공 확률은 없으며, 메인 장비의 레벨과 룬은 그대로 유지된다.

| 현재 등급 | 다음 등급 | 골드 | 재료 수 | 같은 장비 종류 필요 |
|---:|---:|---:|---:|---|
| 0 일반 | 1 고급 | 500 | 3 | 예 |
| 1 고급 | 2 레어 | 900 | 3 | 예 |
| 2 레어 | 3 에픽 | 1,500 | 2 | 예 |
| 3 에픽 | 4 유니크 | 2,400 | 2 | 아니오 |
| 4 유니크 | 5 전설 | 3,600 | 1 | 아니오 |
| 5 전설 | - | 합성 불가 | - | - |

### 선택 규칙

- 메인 장비는 임시장비일 수 없다.
- 메인과 재료는 같은 등급이어야 한다.
- 0~2등급은 같은 장비 종류의 재료가 필요하다.
- 3~4등급은 종류가 달라도 된다.
- 같은 인스턴스를 중복 재료로 쓸 수 없다.
- 메인 장비 자체를 재료로 쓸 수 없다.
- 필요한 재료 수를 실제로 확보할 수 없는 장비는 메인 후보로 잠긴다.

선택한 메인을 다시 누르면 메인과 재료 슬롯을 모두 비운다. 선택한 재료를 다시 누르면 그 재료만 제거하고, 뒤쪽 재료를 앞으로 당긴다.

자동 선택은 메인이 이미 있느냐에 따라 다르게 동작한다.

- 메인이 있으면 유효한 기존 재료를 보존하고 빈자리만 채운다.
- 메인이 없으면 등급 → 레벨 → 장비 종류 → CatalogIndex → ItemId → ItemInstanceId 오름차순으로 가장 낮은 완성 가능한 조합을 찾는다.

합성 화면은 메인을 가상의 다음 등급 장비로 만들어 최대 레벨과 4종 능력치의 결과·상승량을 미리 계산한다. 실제 합성 요청 전에 재료가 완전한지, 최대 등급인지, 골드가 충분한지 다시 검사한다.

현재 `ForgeCombineController`에는 합성 비용 TMP를 갱신하는 배선이 없다. 골드 비용은 실행 직전에 검증하지만, 화면의 비용 숫자를 Title Data와 동기화하려면 별도 표시 연결이 필요하다.

서버는 골드를 차감한 뒤 메인 장비의 등급을 1 올려 저장하고, 마지막으로 재료 인스턴스를 회수한다. 응답 후 클라이언트는 메인 캐시를 패치하고 재료를 제거한 다음 골드 캐시를 차감하며, `Currency.Refreshed`와 `Inventory.Refreshed`를 차례로 발생시킨다.

## 제련

제련은 룬 3개의 옵션과 공통 등급을 다시 추첨하는 종결 성장 콘텐츠다.

제련 화면에 들어가면 필터는 전체, 정렬은 등급 순으로 초기화되고 기존 선택도 비워진다. 임시장비는 목록에서 완전히 제외한 뒤 정렬된 첫 장비를 자동 선택한다. 이후 필터·정렬 또는 `Inventory.Refreshed`로 목록을 다시 만들 때는 같은 `ItemInstanceId`의 새 캐시 객체로 선택을 유지하고, 대상이 사라졌다면 첫 장비로 이동한다. 선택 가능한 장비가 하나 이상인 동안에는 같은 장비를 다시 눌러도 해제되지 않아 항상 하나가 선택된다.

장비가 바뀌면 룬 0번이 자동 선택되고 모든 잠금은 해제된다. 룬도 같은 슬롯을 재클릭하여 선택 해제할 수 없다.

장착 중인 장비도 제련할 수 있으며 인벤토리에는 `E` 표시만 붙는다. 제련 성공 후 새 `EquipmentData` 캐시 객체가 선택에 다시 연결되면 룬 선택은 0번으로, 잠금 상태는 모두 해제로 초기화된다.

### 룬 잠금

- 최대 2개까지 잠글 수 있다.
- 미감정 룬인 `rune_null`은 잠글 수 없다.
- 3개 전부 잠그는 것은 클라이언트와 서버가 모두 거부한다.
- 잠금 버튼과 잠금 해제 버튼은 별도 오브젝트이며 현재 상태에 맞는 하나만 표시한다.

잠금은 **룬 ID를 보존하는 기능**이다. 등급까지 고정하는 기능은 아니다. 제련 결과의 공통 등급이 상승하면 잠긴 룬도 같은 등급으로 함께 오른다.

### 제련 비용

비용 공식은 다음과 같다.

```text
총 비용
= 잠금 개수 기본비
+ 잠긴 슬롯 각각의 등급별 잠금비
+ 재추첨 슬롯 각각의 등급별 재추첨비
```

| 룬 등급 | 재추첨 슬롯 비용 | 잠금 슬롯 등급비 |
|---:|---:|---:|
| 0 | 33,333 | 30,000 |
| 1 | 70,000 | 80,000 |
| 2 | 150,000 | 180,000 |
| 3 | 350,000 | 450,000 |
| 4 | 800,000 | 1,000,000 |
| 5 | 1,800,000 | 2,500,000 |

잠금 개수 기본비는 0개일 때 0, 1개일 때 150,000, 2개일 때 600,000이다.

따라서 미감정 0등급 룬 3개를 처음 제련할 때의 비용은 다음과 같다.

```text
33,333 × 3 = 99,999 골드
```

### 최초 감정 확률

룬 중 하나라도 `rune_null`이면 최초 등급 가중치를 사용한다.

| 결과 등급 | 가중치 | 확률 |
|---:|---:|---:|
| 0 일반 | 5,500 | 55% |
| 1 고급 | 2,800 | 28% |
| 2 레어 | 1,200 | 12% |
| 3 에픽 | 400 | 4% |
| 4 유니크 | 90 | 0.9% |
| 5 전설 | 10 | 0.1% |

이미 일부 높은 등급 룬이 존재한다면 그 등급 미만은 후보에서 제외하고, 남은 가중치를 다시 정규화한다.

### 일반 제련 등급 상승 확률

모든 룬이 감정된 뒤에는 현재 세 룬 중 최고 등급을 기준으로 유지 또는 정확히 한 단계 상승만 추첨한다.

| 현재 등급 | 한 단계 상승 확률 |
|---:|---:|
| 0 일반 | 20% |
| 1 고급 | 10% |
| 2 레어 | 5% |
| 3 에픽 | 2% |
| 4 유니크 | 0.5% |
| 5 전설 | 0% |

결정된 `nextGrade`는 세 슬롯 모두에 적용된다. 잠긴 슬롯은 기존 RuneId를 유지하고, 잠기지 않은 슬롯은 해당 등급의 풀에서 새 RuneId를 뽑는다. 현재 각 등급에는 동일 가중치의 룬 30개가 있으므로 등급이 결정된 뒤 개별 RuneId 확률은 사실상 각각 1/30이다.

등급은 내려가지 않는다. 전설에서는 등급을 유지한 채 잠기지 않은 룬 ID만 다시 추첨한다.

## 장비 초기화

장비 초기화는 현재 등급과 레벨을 기준으로 밸런스 표에서 강화 재화를 역산하고, 장비를 일반 1레벨 상태로 되돌리는 기능이다.

상세 팝업에서 초기화 버튼을 누르면 부모 상세 팝업의 CanvasGroup을 숨기고 전용 초기화 팝업을 연다. 취소하면 상세 팝업을 복원한다. 성공하면 먼저 전체 인벤토리를 다시 읽어 재화와 장비 캐시를 갱신하고, Guard의 최소 입력 차단 시간이 끝난 뒤 초기화 팝업과 상세 팝업을 모두 닫는다.

### 결과 규칙

- 대상 장비의 ItemId와 ItemInstanceId는 유지한다.
- 등급은 0으로 돌아간다.
- 레벨은 1로 돌아가며 UI에는 `+1`로 표시한다.
- 룬 3개는 그대로 유지한다.
- 서버 환급식으로 계산한 골드와 강화석을 전액 지급한다. 이전 등급은 각 등급의 최대 레벨까지 강화했다고 가정한다.
- 합성에 사용한 골드는 환급하지 않는다.
- 임시장비는 초기화할 수 없다.
- 일반 1레벨 장비는 초기화할 수 없다.
- 장착 중인 장비는 UI에서 차단하고 전용 토스트를 표시한다.

장착 여부는 로컬 PlayerPrefs 기반이어서 서버가 알지 못한다. 이 차단은 `EquipmentDetailPopup`에서 LoadoutController를 넘기는 정상 UI 경로의 사용자 보호 정책이며, `EquipmentResetPopup.Show(data)`를 직접 호출하거나 서버 요청을 조작하는 경우까지 보장하는 데이터 무결성 규칙은 아니다.

강화 환급은 이전 등급과 현재 등급을 나누어 계산한다.

```text
이전 등급 0..현재등급-1
  = 각 등급의 (최대 레벨 - 1)회 강화비

현재 등급
  = (현재 레벨 - 1)회 강화비
```

### 합성 재료 환급

재료는 누적 환급이 아니다. **바로 이전 한 단계의 합성 재료만** 같은 부위의 임시장비로 지급한다.

| 초기화 전 등급 | 지급 임시장비 |
|---:|---|
| 0 일반 | 없음 |
| 1 고급 | 0등급 임시장비 3개 |
| 2 레어 | 1등급 임시장비 3개 |
| 3 에픽 | 2등급 임시장비 2개 |
| 4 유니크 | 3등급 임시장비 2개 |
| 5 전설 | 4등급 임시장비 1개 |

환급 미리보기에는 대상 장비, 강화석, 골드, 재료 장비 수와 실제 임시장비 슬롯을 보여준다. 필요한 개수보다 남는 슬롯은 알파만 숨기지 않고 GameObject를 `SetActive(false)`로 꺼서 HorizontalLayoutGroup 정렬에서 완전히 제외한다.

초기화는 새 임시장비를 지급해야 하므로 다른 기능처럼 부분 캐시만 고치지 않는다. 서버 성공 후 전체 UserInventory를 다시 로드하고, 재화와 모든 장비 슬롯이 갱신된 뒤 팝업을 닫는다.

## 임시장비라는 특수 자원

임시장비는 초기화할 때 바로 이전 등급 합성에 필요했다고 간주한 재료를 대체 지급하고, 다시 합성 재료로 사용할 수 있게 만든 장비다.

타입별 ItemId는 다음과 같다.

```text
eq_temp_weapon
eq_temp_helmet
eq_temp_shield
eq_temp_armor
eq_temp_gloves
eq_temp_boots
```

일반 장비와 같은 데이터 구조를 사용하지만, 사용할 수 있는 범위는 명확히 제한했다.

| 기능 | 임시장비 규칙 |
|---|---|
| 기본 인벤토리 | 표시함 |
| 기본 인벤토리 클릭 | 상세 팝업을 열지 않음 |
| 기본 인벤토리 정렬 | 상위 정렬은 따르고, 같은 ID 비교 구간에서 일반 장비 뒤 |
| 장착 | 불가 |
| 강화 | 불가 |
| 합성 메인 | 불가, 클릭 시 전용 토스트 |
| 합성 재료 | 등급·종류 조건이 맞으면 가능 |
| 합성 인벤토리 잠금 | 임시장비에는 잠금 오버레이를 표시하지 않음 |
| 합성 종류 순 | 같은 잠금 상태 그룹 안에서 일반장비 뒤에 배치 |
| 제련 | 목록에서 완전히 제외 |
| 초기화 | 불가 |

합성 화면에서 임시장비는 메인이 될 수 없지만 재료가 될 수 있다. 메인이 비어 있는 상태에서 임시장비를 누르면 선택하지 않고 “임시 장비는 합성 메인 장비로 선택할 수 없습니다.” 토스트를 표시한다. 자동 선택의 재료 후보에도 임시장비가 포함된다.

임시장비는 합성 인벤토리에서 항상 잠금 오버레이를 표시하지 않는다. 그래서 메인과 등급·종류가 맞지 않거나 재료 슬롯이 이미 찬 경우에도 겉으로는 잠기지 않은 채, 클릭하면 재료 등록 없이 조용히 끝날 수 있다. 또한 잠금 상태 비교가 종류 비교보다 우선하므로, 잠긴 일반장비가 있다면 잠기지 않은 임시장비가 그보다 앞에 올 수도 있다. 즉 “일반장비 뒤”는 같은 잠금 그룹 안에서만 보장된다.

## 서버 요청과 화면 새로고침

강화, 합성, 제련, 초기화는 공통으로 다음 경계를 따른다.

```text
UI 사전 검증
→ ServerActionGuard.TryBegin
→ ServerInventoryModule
→ ExecuteCloudScript
→ 서버 재검증과 데이터 변경
→ 로컬 캐시 패치 또는 전체 재로드
→ Currency.Refreshed / Inventory.Refreshed
→ UI 완료 콜백
→ Guard.End 요청
→ 성공·실패 UI 처리
→ 요청 시작 후 최소 0.5초가 지난 시점에 실제 입력 해제
```

클라이언트 검증은 사용자의 불필요한 요청을 막기 위한 것이고, 실제 데이터 보호는 CloudScript가 다시 수행한다. 재료 수, 중복 InstanceId, 각 작업에서 금지된 임시장비 사용, 재화 부족, 룬 잠금 제한 등을 서버에서도 검사한다. 단, 조건에 맞는 임시장비를 합성 재료로 사용하는 것은 허용한다.

강화·합성·제련은 성공 응답의 `after`와 `spent`를 이용해 로컬 캐시를 바로 패치한다. 파싱이나 대상 검색에 실패하면 안전하게 전체 인벤토리를 다시 로드한다. 초기화는 새 아이템 지급이 포함되므로 처음부터 전체 재로드를 사용한다.

`Inventory.Refreshed`의 주요 구독자는 다음과 같다.

- 기본 장비 인벤토리
- 캐릭터 인벤토리
- 합성 인벤토리
- 제련 인벤토리
- 장착 슬롯 컨트롤러

그래서 합성으로 장착 장비가 소모되거나 초기화로 임시장비가 추가되어도 한 이벤트 흐름 안에서 모든 슬롯이 다시 계산된다.

### 입력 차단

서버 변경 버튼에는 `ServerActionGuard`를 둔다.

- 이미 처리 중이면 다음 요청을 거부한다.
- 전역 `GlobalInputBlocker`를 켠다.
- 최소 0.5초 동안 입력 차단을 유지한다.
- 요청이 끝나면 참조 카운트를 줄여 안전하게 해제한다.

Guard가 연결되지 않은 경우에도 각 컨트롤러의 로컬 busy 상태와 GlobalInputBlocker를 사용하는 fallback이 있다. 다만 이 fallback은 콜백 직후 바로 해제되어 Guard의 최소 0.5초 유지 시간은 적용되지 않는다. 최초 서버 로딩 역시 인증, Title Data, Catalog, Inventory 로딩이 끝날 때까지 별도의 전역 차단을 건다.

## 로컬라이제이션과 숫자 표기

텍스트는 거대한 키 테이블 대신 작은 `LocalizedTextData` ScriptableObject를 직접 참조한다. 한 에셋에 한국어, 영어, 일본어를 넣고, 현재 언어 값이 비어 있으면 다른 언어로 fallback한다.

언어 변경에 즉시 반응해야 하는 화면은 `LocalizationRegistry`에 등록한다. 장비명, 등급 배지, 룬 이름과 등급별 설명, 토스트 등이 이 흐름을 사용한다.

현재 남은 제한도 있다. 장비 등급 스킬 이름 5개는 한국어로 하드코딩되어 있고, `CharacterDetailPopup`은 Registry에 등록되지 않아 팝업을 연 채 언어를 바꾸면 이름과 고유능력이 즉시 갱신되지 않는다. 캐릭터 인벤토리의 고유능력 패널 이름도 기존 `DisplayName` 스냅샷이 남을 수 있다.

토스트는 공용 `LocalizedToastMessage`가 담당한다. 같은 메시지가 이미 표시 중이면 중복 재생하지 않고, unscaled time 기준으로 fade in → 유지 → fade out을 실행한다.

재화와 큰 능력치는 `CurrencyFormatter`를 공유한다.

- 1,000 미만: 원본 숫자
- 1,000 이상: k 표기
- 100,000 이상: m 표기
- 반올림이 아니라 자릿수에 맞춰 버림

상단 재화, 연결된 강화·제련 비용, 큰 능력치가 서로 다른 표기법을 사용하지 않도록 통일했다. 합성 비용 표시는 아직 컨트롤러 배선이 남아 있다.

## 반드시 기억해야 할 위험사항

완성된 기능과 별개로, 현재 코드에서 실제로 확인된 위험은 기록해 둬야 한다.

### 1. 합성 후 초기화 환급 악용 가능성

현재 합성에는 “현재 등급의 최대 레벨이어야 한다”는 조건이 없다. 그러나 초기화 환급은 이전 등급을 모두 최대 레벨까지 강화했다고 가정한다.

따라서 낮은 레벨에서 바로 합성한 장비를 초기화하면, 실제로 지불하지 않은 이전 등급 강화비까지 환급될 수 있다. 합성 조건과 환급 가정이 서로 다른 것이 원인이다.

해결 방향은 둘 중 하나다.

- 합성 조건에 현재 등급 최대 레벨을 요구한다.
- 실제 투자 이력을 저장하여 초기화가 그 이력만 환급하게 한다.

### 2. 초기화 밸런스가 이중 소스다

초기화 UI 미리보기는 클라이언트의 강화·합성 Balance를 조합한다. 실제 서버 지급은 별도의 `EquipmentResetBalance`를 읽는다. 현재 로컬 JSON은 서로 일치하지만, 나중에 한쪽만 수정하면 미리보기와 실제 환급이 달라진다.

강화 비용이나 합성 재료 수를 바꿀 때는 Reset Balance도 반드시 함께 수정해야 한다.

### 3. CloudScript는 트랜잭션이 아니다

현재 CloudScript의 여러 단계는 원자적으로 묶이지 않는다.

- 강화: 재화 차감 → 상태 저장
- 합성: 골드 차감 → 메인 저장 → 재료 회수
- 제련: 골드 차감 → 룬 상태 저장
- 초기화: 재화 지급 → 임시장비 지급 → 상태 저장

중간 PlayFab API가 실패하면 앞 단계만 반영될 가능성이 있다. `ok=false` 응답에서는 클라이언트가 자동으로 전체 재로드하지 않으므로, 운영 단계에서는 실패 후 재동기화 정책이나 서버 작업의 멱등성 설계가 필요하다.

### 4. 세션 중 Title Data 변경

클라이언트는 로그인 시 Title Data를 캐시하지만 CloudScript는 요청마다 서버 Title Data를 다시 읽는다. 운영 중 값을 바꾸면 재로그인 전 UI 비용과 실제 서버 비용이 잠시 다를 수 있다.

밸런스 업데이트 시에는 버전 관리, 강제 재로드, 점검 시간 중 반영 중 하나를 선택하는 편이 안전하다.

### 5. 장착 장비의 합성 재료 사용

현재 합성 인벤토리는 장착 여부를 `E`로 표시하지만 재료 사용을 막지는 않는다. 소모 후 장착 슬롯은 자동으로 비워지므로 데이터는 복구되지만, 사용자 실수 방지 정책은 별도로 결정해야 한다.

## 유지보수 체크리스트

### 장비나 룬을 추가할 때

- Catalog의 ItemId와 CustomData를 추가했는가?
- `GlobalGameData`에 아이콘과 이름을 연결했는가?
- 장비 등급 스킬 5개를 채웠는가?
- 룬이라면 이름, 기본 설명, 등급별 설명을 연결했는가?
- 임시장비라면 `eq_temp_부위명` 규칙과 Reset Balance 매핑이 일치하는가?

### 성장 밸런스를 바꿀 때

- 개별 Balance JSON을 수정했는가?
- `TitleDataUpload.json`도 다시 만들었는가?
- 초기화 Balance의 복제 값도 함께 바꿨는가?
- 클라이언트 미리보기와 CloudScript 실제 결과를 같은 장비로 비교했는가?
- 현재 로그인 세션의 Title Data 캐시를 다시 읽었는가?

### 서버 변경 로직을 바꿀 때

- UI 사전 검증만이 아니라 CloudScript 검증도 추가했는가?
- 성공 응답의 `after`, `spent` 구조가 클라이언트 파서와 일치하는가?
- 캐시 패치 실패 시 전체 재로드 fallback이 있는가?
- `Inventory.Refreshed`, `Currency.Refreshed`가 필요한 횟수만 발생하는가?
- 버튼에 `ServerActionGuard`가 연결되어 있는가?
- 실패 토스트의 LocalizedTextData가 씬에 연결되어 있는가?

### 팝업과 씬을 바꿀 때

- `OnEnable`에서 등록한 버튼·이벤트를 `OnDisable`에서 해제하는가?

- 배열 슬롯 수와 실제 UI 슬롯 수가 일치하는가?
- HorizontalLayoutGroup의 빈 슬롯은 필요하다면 GameObject까지 비활성화하는가?

## 마무리

이번 작업에서 가장 어려웠던 것은 기능 하나하나가 아니었다. **같은 장비가 기본 인벤토리, 장착 슬롯, 합성, 제련, 초기화에서 서로 다른 규칙으로 보이면서도 결국 같은 서버 상태를 가리켜야 한다는 것**이 핵심 난제였다.

그래서 공용 셀을 만들고, 데이터는 서버 캐시에 모으고, 화면별 선택 규칙은 컨트롤러에 분리하고, 변경이 끝나면 이벤트로 다시 모으는 구조를 택했다. 임시장비처럼 예외가 많은 데이터도 각 화면의 역할에 맞게 명시적인 정책을 부여했다.

이제 인벤토리 탭은 단순히 “아이템이 보이는 화면”이 아니다. 장비와 캐릭터의 소유, 성장, 조합, 선택, 환급을 하나의 규칙 아래 묶는 기반 시스템이 되었다.

오랫동안 붙잡고 있던 만큼 마무리할 때 내는 소리도 정해져 있다.
